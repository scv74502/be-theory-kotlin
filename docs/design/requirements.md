## 2. 서비스 도메인 용어 정의

**도메인(Domain)이란?**

도메인은 **소프트웨어 서비스로 해결하고자 하는 사용자의 문제 영역이자 비즈니스 활동의 범위**다.
서비스가 시장에서 가치를 창출하는 가장 핵심적인 활동이 무엇인지 정의한 영역이며, 이 범위 내에서 활동하는 주체(Actor)와 행동(Behavior)들의 유기적인 관계를 포함한다.

**도메인 용어(Domain Language)란?**

기획자, 개발자, 사업 담당자 등 직군에 상관없이 서비스 내에서 자주 사용되고 비즈니스에 핵심적인 용어를 정의한 것이다.
용어 종류와 개념이 확립되지 않으면 같은 단어라도 서비스의 성격에 따라 용어의 정의나 범위, 역할이 상이할 수 있다.
또한 구성원마다 동일한 단어를 다르게 해석하여 논의와 작업 결과물의 정합성이 깨질 수 있다.
따라서 서비스 내의 명확한 기준으로 용어의 개념을 선언하여 이를 통해 **유비쿼터스 언어(Ubiquitous Language)**를 확립하고, 직군 간·개인 간의 개념을 일치시켜 프로젝트의 방향성과 구현 모델을 통일하는 효과를 얻는다.

### 도메인 용어 목록

#### 사용자

서비스를 이용하는 회원이다. 사용자는 회원가입을 통해 생성되며, 로그인 ID와 비밀번호를 통해 '로그인 필요 API'의 요청자로 식별된다.
서비스 대고객 회원인 '일반 사용자' 혹은 '사용자'와 '일반 사용자'와 '상품', '브랜드' 등 일반 사용자의 서비스 이용을 보조하는 '관리자' 두 유형으로 구분된다.

##### 값과 규칙

- 로그인 ID: 영문과 숫자로 이루어진 4~20자 문자열이며, 시스템 전체에서 유일해야 한다.
- 비밀번호: 8~16자 문자열이며, 영문 대문자, 영문 소문자, 숫자, 특수문자를 각각 1개 이상 포함해야 한다. 생년월일 토큰(`yyyyMMdd`, `yyMMdd`, `MMdd`)은 포함할 수 없으며, 저장 시에는 단방향으로 인코딩된 값만 보관한다.
- 이름: 공백이 아닌 1~50자 문자열이다.
- 생년월일: 과거 날짜만 허용한다.
- 이메일: 표준 이메일 형식을 따른다.

##### 행위 정의

- 회원가입: 서비스에 가입되지 않은 사용자가 서비스 이용에 필요한 개인정보와 식별자를 등록하여 계정을 새로 생성하는 행위다.
- 사용자 정보 조회: 로그인한 사용자가 자신의 계정 식별자와 암호를 바탕으로 자신의 정보를 조회하는 행위다.
- 비밀번호 변경: 로그인한 사용자가 현재 비밀번호를 검증한 뒤, 사용중인 현재 비밀번호와는 다른 새 비밀번호로 변경하는 행위다.

#### 브랜드

브랜드는 어떤 경제적인 생산자를 구별하는 지각된 이미지와 경험의 집합이며 보다 좁게는 어떤 상품이나 회사를 나타내는 상표, 표지이다. 
이 서비스에서는 주로 상품이 소속된 그룹이나 집합, 태그의 역할을 수행하며 상품은 특정 브랜드에 소속될 수 있다.
사용자는 브랜드 정보를 조회할 수 있고, 관리자는 브랜드를 등록하고 관리한다.

##### 값과 규칙

- 브랜드 식별자: 브랜드를 조회하거나 상품을 필터링할 때 사용하는 식별 값이다.
- 브랜드와 상품의 관계: 상품은 이미 시스템에 등록된 브랜드에만 연결될 수 있다.
- 브랜드 삭제: 브랜드가 삭제되면 해당 브랜드에 속한 상품들도 함께 영속적으로 삭제되어야 한다.

##### 행위 정의

- 브랜드 조회: 사용자가 특정 브랜드 정보를 조회하는 행위다.
- 브랜드 목록 조회: 관리자가 등록된 브랜드 목록을 조회하는 행위다.
- 브랜드 등록: 관리자가 새로운 브랜드를 시스템에 등록하는 행위다.
- 브랜드 정보 수정: 관리자가 등록된 브랜드 정보를 변경하는 행위다.
- 브랜드 삭제: 관리자가 등록된 브랜드와 그 브랜드에 속한 상품들을 삭제하는 행위다.

#### 상품

사용자가 탐색하고 좋아요를 표시하거나 주문할 수 있는 판매 대상이다. 상품은 하나의 브랜드에 속하며, 주문 시점에는 상품의 주요 정보가 주문 스냅샷으로 보관된다.

##### 값과 규칙

- 상품 식별자: 상품 상세 조회, 좋아요, 주문 항목에서 상품을 가리키는 식별 값이다.
- 브랜드: 상품은 등록된 브랜드에 속해야 하며, 한 번 등록된 상품의 브랜드는 임의로 수정할 수 없다.
- 상품명과 가격: 주문 시점에 주문 스냅샷으로 보관되어야 하는 대표 정보다.
- 재고: 상품의 주문 가능 여부를 결정하는 수량이며, 주문 시점에 확인 및 차감되어야 한다.
- 목록 조회 조건: 브랜드 식별자로 필터링할 수 있고, 최신순(`latest`) 정렬은 필수로 제공한다. 가격 낮은 순(`price_asc`)과 좋아요 많은 순(`likes_desc`)은 선택적으로 확장할 수 있다.
- 페이지 조건: 상품 목록은 페이지 번호와 페이지당 상품 수를 기준으로 조회한다.

##### 행위 정의

- 상품 목록 조회: 사용자가 조건에 맞는 상품 목록을 조회하는 행위다.
- 상품 상세 조회: 사용자가 특정 상품 정보를 조회하는 행위다.
- 상품 등록: 관리자가 등록된 브랜드에 속한 상품을 새로 등록하는 행위다.
- 상품 정보 수정: 관리자가 등록된 상품 정보를 변경하는 행위다.
- 상품 삭제: 관리자가 등록된 상품을 삭제하는 행위다.

#### 좋아요

사용자가 특정 상품에 관심을 표시한 상태다. 좋아요는 사용자와 상품 사이의 관계로 표현되며, 사용자는 자신이 좋아요 한 상품 목록을 조회할 수 있다.

##### 값과 규칙

- 사용자: 좋아요를 등록하거나 취소하는 주체다.
- 상품: 좋아요의 대상이다.
- 좋아요 목록: 로그인한 사용자가 좋아요 한 상품들의 목록이다.

##### 행위 정의

- 좋아요 등록: 로그인한 사용자가 특정 상품에 관심을 표시하는 행위다.
- 좋아요 취소: 로그인한 사용자가 특정 상품에 대한 관심 표시를 해제하는 행위다.
- 좋아요 목록 조회: 로그인한 사용자가 자신이 좋아요 한 상품 목록을 조회하는 행위다.

#### 쿠폰

사용자가 발급받아 주문에 사용할 수 있는 혜택이다. 현재 요구사항에서는 서비스 흐름에 포함되는 도메인으로 정의하며, 할인 방식과 사용 조건은 추후 상세 요구사항에서 확장한다.

##### 값과 규칙

- 쿠폰 소유자: 쿠폰을 발급받은 사용자다.
- 쿠폰 혜택: 주문에 적용될 수 있는 할인 또는 보상 정보다. 할인 금액, 할인율, 유효기간, 사용 상태는 추후 확장 가능하도록 둔다.

##### 행위 정의

- 쿠폰 발급: 사용자가 주문 전에 혜택을 받는 행위다.
- 쿠폰 사용: 주문 과정에서 발급받은 쿠폰 혜택을 적용하는 행위다. 구체적인 사용 규칙은 추후 상세 요구사항에서 정의한다.

#### 주문

사용자가 상품을 구매하는 과정과 그 결과물을 말한다. 주문은 사용자와 상품을 연결하며, 주문 요청 시 상품 재고 확인 및 차감이 정상적으로 보장되어야 한다.

##### 값과 규칙

- 주문자: 주문을 요청한 로그인 사용자다.
- 주문 기간: 주문 목록 조회 시 시작일과 종료일을 기준으로 조회할 수 있다.
- 주문 상세: 단일 주문의 결과와 포함된 주문 항목을 확인할 수 있는 정보다.
- 재고 정합성: 주문 시점에 상품 재고를 확인하고 차감해야 한다.
- 결제 연동: 결제 기능은 주문 프로세스의 기본 동작을 안정화한 후 추가로 개발한다.

##### 주문 항목

주문에 포함된 개별 상품 구매 단위다. 하나의 주문은 하나 이상의 주문 항목을 가질 수 있고, 각 주문 항목은 주문할 상품과 수량을 가진다.

###### 값과 규칙

- 상품 식별자: 주문 항목이 어떤 상품을 구매하려는지 나타내는 값이다.
- 수량: 해당 상품을 몇 개 주문할지 나타내는 값이다.
- 주문 시점 정보: 주문 항목은 주문 스냅샷을 통해 주문 당시의 상품명, 가격 등의 정보를 보존할 수 있다.

###### 행위 정의

- 주문 항목 구성: 사용자가 주문 요청 본문에 상품 식별자와 수량을 담아 구매 대상을 구성하는 행위다.

##### 행위 정의

- 주문 요청: 로그인한 사용자가 하나 이상의 상품과 수량을 지정해 구매를 요청하는 행위다.
- 주문 목록 조회: 로그인한 사용자가 자신의 주문 목록을 조회하는 행위다.
- 주문 상세 조회: 로그인한 사용자가 자신의 특정 주문 상세를 조회하는 행위다.
- 주문 관리자 조회: 관리자가 주문 목록 또는 단일 주문 상세를 조회하는 행위다.

#### 결제

주문에 대한 지불 처리다. 현재 요구사항에서는 주문 프로세스의 기본 동작을 안정화한 뒤 연동할 확장 도메인으로 정의한다.

##### 값과 규칙

- 결제 대상: 결제는 특정 주문을 대상으로 한다.
- 결제 확장 정보: 결제 수단, 결제 상태, 결제 금액은 추후 요구사항에서 확장할 수 있도록 둔다.

##### 행위 정의

- 결제 연동: 주문 프로세스가 안정화된 뒤 외부 또는 내부 결제 처리와 연결하는 행위다.

#### 재고

상품의 주문 가능 수량이다. 재고는 주문 가능 여부를 판단하는 기준이며, 주문 요청이 처리될 때 확인되고 차감되어야 한다.

##### 값과 규칙

- 상품: 재고가 귀속되는 대상이다.
- 주문 가능 수량: 주문 시점에 구매 가능한 상품 수량이다.
- 정합성: 주문 시점의 재고 확인과 차감은 정상적으로 보장되어야 한다.

##### 행위 정의

- 재고 확인: 주문 요청 시 상품을 주문할 수 있는 수량이 남아 있는지 확인하는 행위다.
- 재고 차감: 주문이 처리될 때 주문 수량만큼 상품 재고를 줄이는 행위다.

#### 주문 스냅샷

주문 시점의 상품 정보를 주문에 보관한 기록이다. 상품 정보가 이후 변경되더라도 과거 주문 내역은 주문 당시의 정보를 기준으로 유지되어야 한다.

##### 값과 규칙

- 상품명: 주문 시점의 상품 이름이다.
- 가격: 주문 시점의 상품 가격이다.
- 보존 기준: 주문 이후 상품명이나 가격이 변경되더라도 기존 주문 내역은 훼손되지 않아야 한다.
- 확장 정보: 주문 당시 필요한 상품 노출 정보는 추후 요구사항에 따라 함께 보관할 수 있다.

##### 행위 정의

- 주문 스냅샷 생성: 주문이 생성될 때 주문 당시의 상품 정보를 함께 저장하는 행위다.
- 주문 스냅샷 조회: 주문 상세 조회에서 주문 당시의 상품 정보를 확인하는 행위다.

## 3. API 설계 제안 및 인증 가이드

### API Endpoint Prefix
* **대고객 서비스**: `/api/v1` prefix를 통해 제공된다.
* **관리자(Admin) 서비스**: `/api-admin/v1` prefix를 통해 제공된다.

### 인증 및 사용자 식별 방식
본 프로젝트는 학습용 프로젝트이므로 복잡한 실 인증/인가(Spring Security, JWT 등)는 연동하지 않는다. 대신 아래의 **HTTP 헤더**를 활용하여 요청자를 식별한다.

> [!IMPORTANT]
> - 인증/인가 로직 자체는 주요 범위가 아니므로 명세대로 헤더 파싱 형태로만 구현한다.
> - 유저는 타 유저의 정보에 직접 접근할 수 없도록 철저히 검증 및 제어해야 한다.

#### 대고객 기능 인증 헤더 (로그인 필요 API에 사용)
| HTTP Header Key | Description | Example |
| :--- | :--- | :--- |
| **`X-Loopers-LoginId`** | 로그인 ID | `user123` |
| **`X-Loopers-LoginPw`** | 비밀번호 | `securepass` |

#### 어드민 기능 인증 헤더 (관리자 전용 API에 사용)
| HTTP Header Key | Description | Example |
| :--- | :--- | :--- |
| **`X-Loopers-Ldap`** | 관리자 식별용 LDAP 정보 | `loopers.admin` |

| `GET` | `/api/v1/products/{productId}` | X | 상품 정보 조회 |

#### 상품 목록 조회 쿼리 파라미터 (Query Parameters)
| Parameter | Example | Default Value | Description |
| :--- | :--- | :---: | :--- |
| `brandId` | `1` | - | 특정 브랜드의 상품만 필터링 |
| `sort` | `latest` | `latest` | 정렬 기준 (`latest` / `price_asc` / `likes_desc`) |
| `page` | `0` | `0` | 페이지 번호 |
| `size` | `20` | `20` | 페이지당 상품 수 |

> [!TIP]
> **정렬 기준(sort) 구현 가이드:**
> * 정렬 기준 구현 범위는 유연하게 설계하되, **최신순(`latest`)**은 필수적으로 구현한다.
> * 그 외 가격 낮은 순(`price_asc`), 좋아요 많은 순(`likes_desc`) 정도는 상황에 따라 선택적으로 추가 구현한다.

#### 주문 요청 (POST `/api/v1/orders`) Body 예시
```json
{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ]
}
```

> [!IMPORTANT]
> **주문 처리 핵심 비즈니스 룰:**
> 1. **결제 연동**: **결제** 기능은 주문 프로세스의 기본 동작을 안정화한 후, **추가로 개발**을 이어간다.
> 2. **데이터 스냅샷**: **주문 정보**에는 주문 시점의 상품명, 가격 등의 정보를 **스냅샷** 형태로 보관하여 상품 가격이 변하더라도 과거 주문 내역이 훼손되지 않아야 한다.
> 3. **재고 정합성**: 주문 시점에 **상품 재고 확인 및 차감이 정상적으로 보장**되어야 한다.

## 5. 심화 과정

> [!TIP]
> **성장과 심화를 위한 탐색 주제:**
> 모든 기능의 정상적인 비즈니스 로직 구현을 완료한 후, 실제 대규모 상용 서비스에서 마주칠 수 있는 복잡한 문제들을 깊이 탐구하고 해결 방법을 모색해 본다.
> 
> * **동시성 제어 (Concurrency Control)**: 동일한 상품에 동시 다발적인 대량의 주문 및 재고 차감 요청이 인입될 때 데이터의 정확성을 보장하는 방법 (낙관적/비관적 락, 분산 락 등)
> * **멱등성 보장 (Idempotency)**: 동일한 주문 요청 네트워크가 중복해서 인입되는 경우에 대한 안전 장치 설계
> * **분산 시스템 데이터 일관성 (Consistency)**: 여러 서비스 간 트랜잭션 롤백 및 보상 트랜잭션 기법 연구
> * **조회 성능 최적화 (Slow Query Optimization)**: 상품 목록 필터링 및 대량의 주문 목록 조회 쿼리에 대한 최적의 인덱스 설계와 튜닝
> * **동시 대량 주문 처리**: 핫스팟 상품에 대한 선착순 주문 및 트래픽 폭주 대응 방안
