# VCS Rules

이 문서는 Git/GitHub 기반 작업 흐름을 정의한다.
개발 방식, DDD 경계, 테스트 작성 규칙은 `agent_rules/groundRules.md` 를 따른다.

## 적용 범위

- 현재 브랜치의 기능 개발, 버그 수정, 리팩토링
- 이전 브랜치의 PR 리뷰 수정 및 픽스 작업
- CodeRabbit, Copilot, 사람 리뷰 등 GitHub PR 코드리뷰 대응
- CI 실패 대응, PR/Issue 갱신, 커밋 분리, worktree 사용
- 개발 주차별 브랜치 달성목표 확인 및 현재 브랜치 작업상태 평가

## 기본 원칙

- 현재 브랜치와 리뷰 대상 브랜치가 다르면 변경을 섞지 않는다.
- 리뷰 수정은 원칙적으로 리뷰 대상 브랜치 또는 별도 worktree 에서 수행한다.
- 작업트리에 있는 사용자 변경은 되돌리지 않는다.
- 구현 변경과 리뷰 대응 작업은 같은 커밋/PR에 섞지 않는다.
- PR 본문, GitHub 댓글, 리뷰 스레드 resolve, 이슈 체크박스 갱신 같은 원격 쓰기 작업은 사용자가 요청하거나 승인한 경우에만 수행한다.
- 커밋은 작업 단위에 맞춰 분리한다. TDD Phase 는 Red 를 먼저 수행하되, 커밋은 테스트와 구현을 같은 작업 단위로 묶을 수 있다.
- 저장소 루트 기준 `docs/quest-per-week/` 는 개발 주차별 브랜치 달성목표와 체크리스트의 위치다. 현재 브랜치의 진행 상태를 평가하거나 다음 작업 범위를 정할 때 해당 주차 문서를 함께 확인한다.

## 작업 시작 순서

1. 현재 브랜치와 작업트리 상태를 확인한다.
   - `git status --short --branch`
2. 현재 브랜치의 개발 주차 또는 요청 범위에 해당하는 `docs/quest-per-week/` 문서를 확인한다.
3. 작업 대상이 현재 브랜치인지, 이전 PR 리뷰 수정인지 구분한다.
4. 이전 PR 리뷰 수정이면 upstream PR 번호, head branch, base branch, origin/upstream 관계를 확인한다.
5. upstream PR 의 실제 코드리뷰 코멘트를 다시 조회한다.
6. origin fork 의 리뷰 대응 이슈 또는 체크리스트를 확인한다.
7. 현재 브랜치와 리뷰 대상 브랜치가 다르면 사용자 승인 없이 현재 브랜치에 수정하지 않는다.
8. 필요하면 리뷰 대상 브랜치로 전환하거나 별도 worktree 를 만든다.

## 이전 PR 리뷰 수정 흐름

1. upstream PR 링크 또는 번호를 확인한다.
2. PR head branch 와 로컬 브랜치가 일치하는지 확인한다.
3. 일치하지 않으면 아래 중 하나를 선택한다.
   - 리뷰 대상 브랜치로 전환한다.
   - `git worktree` 로 별도 작업 디렉터리를 만든다.
   - 사용자에게 현재 브랜치에서 진행해도 되는지 확인한다.
4. CodeRabbit, Copilot, 사람 리뷰 코멘트를 upstream PR 기준으로 재조회한다.
5. origin fork 에 단일 GitHub Issue 를 만들고 각 지적사항은 체크리스트로 관리한다.
6. 구현은 Red > Green > Blue 순서로 진행한다.
7. 필요한 테스트와 `ktlintCheck` 를 실행한다.
8. 변경 파일, 검증 결과, 남은 리뷰 항목을 보고한다.

## 커밋 규칙

- Red Phase 테스트는 먼저 작성하고 의도한 실패를 확인한다.
- 같은 작업 단위의 Red Phase 테스트, Green Phase 구현, Blue Phase 리팩토링은 하나의 커밋으로 묶을 수 있다.
- 서로 다른 리뷰 지적사항은 가능하면 커밋을 분리한다.
- 커밋 메시지는 제목 한 줄로만 작성하고 별도 본문을 쓰지 않는다.
- 커밋 메시지 제목에 `[prefix]` 형식의 접두어를 붙이고, `[feat]`, `[agent]`, `[docs]`, `[fix]`, `[refactor]` 로 한정한다.
- 도메인명, 패키지명, 작업 대상명은 `[product]`, `[order]` 같은 접두어로 쓰지 않고 제목 본문에 포함한다.
- 커밋 전에는 staged 파일이 의도한 작업 범위에만 속하는지 확인한다.
- 사용자가 요청하지 않은 문서, 포맷, 리팩토링 변경은 커밋에 포함하지 않는다.

## worktree 사용 기준

- 현재 브랜치에 미완료 작업이 있고 이전 PR 리뷰 수정을 해야 하면 worktree 를 우선 검토한다.
- 리뷰 대상 브랜치와 현재 브랜치가 다르고 변경 범위가 섞일 위험이 있으면 worktree 를 사용한다.
- worktree 생성 전에는 대상 브랜치와 경로를 사용자에게 명확히 보고한다.

## 원격 작업 규칙

- PR body 수정, issue 생성/수정, review thread resolve, comment 작성은 원격 쓰기 작업으로 취급한다.
- 원격 쓰기 작업은 사용자 요청 또는 승인 후 수행한다.
- 원격 작업 후에는 다시 조회하여 반영 여부를 확인한다.
- PR 작업목록은 PR 본문에 직접 누적하지 않고, origin fork 의 GitHub Issue 에 관리한다.
